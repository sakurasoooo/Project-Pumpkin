> 声明：直接编辑 Unity 的 `.unity` 和 `.prefab` 文件属于“高危操作”。本技能要求在编辑前必须完全理解 UnityYAML 序列化协议和对象拓扑结构，拒绝依赖“试试看能不能加载”的黑盒试错。

1. 核心定义
编辑 UnityYAML 文件的核心在于将其视为一张有向图（对象和组件的引用关系），而非普通的文本文档。核心准则包括：
- **拓扑透明**：任何新增或删除组件的操作，必须同时维护父节点（如 `GameObject`）的 `m_Component` 数组和子节点（如 `Transform`）的引用，形成闭环。
- **格式绝对服从**：严格遵守 UnityYAML 的子集规范——不支持注释、不支持原生态 YAML 的 Chomping 符号（+、|）、不支持复杂映射键。
- **确定性修改**：避免可能触发 Unity Asset 重新序列化引起大面积 Diff 的模糊修改，严格控制缩进和换行。

2. 古法编辑的现实价值
- **无损冲突解决**：在 Git 合并 Scene/Prefab 冲突时，通过解析 `fileID` 和引用关系，手动拼接出正确的对象树，而不依赖极易出错的外部文本合并工具。
- **突破编辑器限制**：处理某些在 Unity Editor 中因脚本丢失、循环引用或数据损坏而无法打开（卡死）的场景文件。
- **极客自动化**：通过脚本或正则进行底层属性的精准批量替换，而无需启动庞大且耗时的 Unity 引擎。

3. 古法编程的实践维度 (UnityYAML 专用)

| 维度 | 黑盒盲猜模式 | 古法白盒模式 |
|------|------------|--------------|
| **数据理解** | 当成普通文本全局搜索替换 | 识别 `--- !u![ClassID] &[fileID]` 建立精准对象映射 |
| **浮点数修改** | 随意修改导致浮点数解析错误或精度丢失 | 严格遵守文档：移除 `0x` 前缀和括号里的值，只输入纯十进制数（如 `1.000`） |
| **增加节点** | 随便找个位置粘贴 YAML 块 | 精心分配未被占用的 `fileID`，并在相关 `GameObject` 中正确注册引用 |
| **序列结构** | 使用短划线 `-` 随便写数组结构 | 确保所有序列（如 `m_Component`）都从属于一个 Mapping 的 Value，且严格缩进 |
| **多行文本** | 随意换行 | 多行 plain scalar 必须比上一行有更多缩进，防止引发后续语法解析崩溃 |
| **验证方式** | 打开 Unity 引擎看是否报错 | 通过静态逻辑检查确认 `fileID` 是否存在悬空引用，YAML 格式是否越界 |

4. 如何指导 AI 进行“古法风格”的 UnityYAML 编辑
如果你需要 AI 辅助你修改 Scene/Prefab 文件，你应该这样下达 Prompt：

- **强制依赖拓扑图**
  “在修改这个 Transform 的属性前，请先向我证明你找到了它的父 GameObject，并列出它的 `m_GameObject` 引用的 fileID。”
- **严格的浮点数安全审计**
  “我要修改这个 LocalPosition 的坐标值。请不要试图生成带有 `0x` 前缀的十六进制浮点数，请直接输出十进制数字，并向我解释为什么这样做符合 Unity 官方文档的安全建议。”
- **孤立节点检查模式**
  “这是我尝试合并后的 prefab 文件片段。请充当底层序列化引擎，检查是否存在被 GameObject 移除了 `m_Component` 引用，但仍然残留在文件末尾的僵尸组件节点。”

# 精准替换模式 (Precise Patch Mode)
**旨在改变**：AI 随意重写导致整个 YAML 文件格式混乱、触发引擎大面积重写（Dirty）的现象。

**指令**：“在提供修改建议时，必须严格只输出需要变化的行及其上下文（Diff 格式）。如果是多行文本（Multi-line scalar），必须严格遵守比上一行缩进更多的规则。严禁改变未涉及属性的缩进，严禁添加任何 YAML 注释。”

# 引用溯源反问模式 (Reference Tracing Mode)
**旨在改变**：只改其一不顾全局的破坏性修改。

**生效时机**：准备删除或新增组件数据时

**指令**：“在提出删除或替换某块 `--- !u!xxx &yyy` 序列化节点之前，必须反问我：‘我已经检查了全局上下文，该 `fileID` (yyy) 在哪些地方被引用，我们是否需要一并清理或更新这些引用？’ 只有在确认清理了所有悬空引用后，修改才是合法的。”
